# 🏛️ THE v11A3 MASTER BLUEPRINT: RELEASE CANDIDATE ZERO

## DIRECTIVE 1: The "Header-Only Lie" & Linkage Architecture
**The Flaw:** `src/mathlib.c` is 530 bytes and just `#include`s headers. Because everything is `static inline`, `libmathc.a` exports almost zero symbols. You are pretending to be a static library, but you are actually a header-only library.
**The Fix:** We must commit to the architecture. Since MathLib is heavily focused on zero-alloc/embedded/SIMD inlining, **Header-Only is the correct architectural choice**, but the build system must reflect it.

| File | Context | The Exact Fix |
| :--- | :--- | :--- |
| `CMakeLists.txt` | `add_library(mathc STATIC src/mathlib.c)` | Change to `add_library(mathc INTERFACE)`. Remove `src/mathlib.c` entirely. Change `target_link_libraries` to `target_link_libraries(mathc INTERFACE m)`. |
| `src/mathlib.c` | Entire file | **DELETE THIS FILE.** It is dead weight and confuses linkers. |
| `Makefile` | `SRC = src/mathlib.c` | Remove `SRC`. Change test targets to compile directly against headers: `$(CC) $(CFLAGS) -Iinclude/mathlib -o test tests/test.c $(LDLIBS)` |

## DIRECTIVE 2: Microscopic Numerical Safety (Kimi & GLM's Hitlist)
**The Flaw:** The fuzzer passes because it doesn't hit the exact bit-patterns that cause silent corruption, stack overflows, or undefined behavior.
**The Fix:** Surgical corrections to 8 specific mathematical edge cases.

| File | Function | The Flaw | The Exact Fix |
| :--- | :--- | :--- | :--- |
| `ml_core.h` | `ml_ldexp_pure` | `cast.u += ((uint64_t)exp << 52);` If `exp` is negative, casting to `uint64_t` creates a massive number, corrupting the sign/mantissa bits. | Cast to `int64_t` first, then mask: `cast.u = (cast.u & 0x800FFFFFFFFFFFFFULL) | (((uint64_t)((int64_t)exp + 1023)) << 52);` (Requires extracting mantissa first). |
| `ml_trig.h` | `ml_tan` | `if (c == 0.0) return (s > 0) ? 1.0/0.0 : -1.0/0.0;` If `s == 0.0` and `c == 0.0`, it returns `-inf` instead of `NaN`. | `if (c == 0.0) return (s == 0.0) ? 0.0/0.0 : ((s > 0) ? 1.0/0.0 : -1.0/0.0);` |
| `integral.h` | `gamma_new` | Recursive without depth limit. `gamma_new(10000)` causes a stack overflow. | Cap at `171.0` (where `double` overflows to `inf`): `if (x > 171.0) return 1.0/0.0;` Convert recursion to an iterative `while` loop. |
| `integral.h` | `integral_traditional` | Silent truncation. If `max_steps` is hit, it returns a partial, wrong result. | Change the `break;` to `return 0.0/0.0;` (NaN) to explicitly signal failure when the temporal ceiling is hit. |
| `ml_core.h` | `ml_fmod` | `if (q > 9.22e18) q = 9.22e18;` Casting `9.22e18` to `long long` is dangerously close to `LLONG_MAX` and risks UB. | Use a `while` loop subtraction for quotients `> 1e15`, or use `int64_t` safely with explicit bounds checking. |
| `ml_complex.h` | `cplx_div` | `denom = b.real*b.real + b.imag*b.imag;` Underflows to `0.0` for subnormals, returning `NaN` instead of `Inf`. | Implement **Smith's Method**: Scale by `max(|real|, |imag|)` before division to prevent subnormal underflow/overflow. |
| `ml_trig.h` | `ml_atan2` | Fails to distinguish `+0.0` and `-0.0`. `atan2(+0.0, -0.0)` should be `pi`, but `x == 0.0` catches both. | Use bitwise sign extraction or `ml_copysign` to check the sign bit of `x` and `y` explicitly. |
| `combinatorics.h`| `ncr` | `den1 * den2` can overflow `uint64_t` even if the final mathematical result fits. | Use the iterative multiplicative formula: `result = result * (n - i) / (i + 1)` to prevent intermediate overflow. |

## DIRECTIVE 3: Build System & Tooling Hygiene
**The Flaw:** The build system is a mix of GNU Make quirks and CMake inconsistencies that will break on Windows (Yellow Team) or cause silent rebuild failures.

| File | Context | The Exact Fix |
| :--- | :--- | :--- |
| `Makefile` | `LDFLAGS = -lm` | Change to `LDLIBS = -lm`. (`LDFLAGS` is for `-L` paths, `LDLIBS` is for `-l` libraries). |
| `Makefile` | Missing dependency tracking | Add `CFLAGS += -MMD -MP` and `-include $(DEPS)` so modifying a header actually triggers a recompile. |
| `.gitignore` | `Makefile` | **Remove `Makefile` from `.gitignore`**. Ignoring it causes massive confusion when CMake generates an in-source `Makefile` that overwrites the hand-written one. |
| `CMakeLists.txt` | `set(CMAKE_C_STANDARD 99)` | Add `set(CMAKE_C_EXTENSIONS OFF)`. This prevents GCC/Clang from silently allowing `typeof` or statement expressions. |
| `CMakeLists.txt` | Missing `find_package` support | Add `include(CMakePackageConfigHelpers)` and generate `mathlibConfig.cmake` so downstream users can actually use `find_package(mathlib)`. |
| `compat.h` | `#warning "..."` | `#warning` is a GCC extension and breaks MSVC. Change to `#pragma message("MathLib V1.1: Using deprecated V1.0 API...")`. |

## DIRECTIVE 5: The Silicon Proof (Gemini's Mandate)
**The Flaw:** We claim Horner's method is fast, but we haven't verified that the compiler isn't spilling registers or failing to emit FMA instructions.

**The Action Plan:**
1. **The FMA Audit:** Create `validation/fma_audit.c`. Compile it with `gcc -O3 -mfma -S -fno-fast-math -ffp-contract=off`.
2. **The Grep:** Run `grep -c "vfmadd" fma_audit.s`. If the count is `0`, the compiler is respecting `ffp-contract=off` too strictly and falling back to `mulsd` + `addsd`.
3. **The Fix:** If FMA is missing, we must explicitly use `__builtin_fma(a, b, c)` inside the Horner loops in `ml_exp_log.h` and `internal/minimax.h` to force the hardware instruction without enabling global fast-math.
4. **The AVX2 Magic Constant Audit:** Kimi noted the `0x5fe6ec85e7de30daLL` constant in `simd_bare_metal.h` might just be a blindly ported float constant. Blue team must mathematically verify this constant is optimal for 64-bit `double` precision, or derive the correct one.
