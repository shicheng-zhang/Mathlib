Critical issue #1:
Define the "kernel"

Right now I would ask:

    "What is the smallest possible piece of mathlib that everything else depends on?"

That is your kernel.

It should probably be:

ml_core
|
├── ml_types
├── ml_config
├── ml_error
├── ml_precision
└── ml_platform

Everything else depends downward.

Never sideways.

Bad:

Good:

          applications

               |

        numerical modules

               |

          math primitives

               |

            core layer

Your dependency graph should look like a tree.
Critical issue #2:
Stop treating all accuracy equally

This is probably the biggest numerical design issue.

A professional library does not say:

    "Everything is maximum precision."

That is inefficient.

You need explicit modes.

Something like:

typedef enum
{
    ML_ACCURACY_FAST,
    ML_ACCURACY_BALANCED,
    ML_ACCURACY_STRICT
} ml_accuracy_mode;

Then:

ml_sin(x, ML_ACCURACY_STRICT);

or preferably:

ml_context_t ctx;

ctx.accuracy = ML_STRICT;

ml_sin(&ctx,x);

Why?

Because:

A robotics controller.
A video game.
A physics simulation.
A financial model.

Do not want the same thing.

2. Header hygiene

I would inspect every public header.

The rule:

A public header should answer:

    "What can a user rely on?"

Nothing else.

Bad:

#define ML_INTERNAL_PI 3.141592653589793

in a public header.

Why?

Now users depend on it.

Better:

#define ML_PI 3.14159265358979323846

if public.

or:

static const double ml_pi_internal;

if internal.

I would check every header for:
Missing include guards

Every header:

#ifndef ML_TRIG_H
#define ML_TRIG_H

...

#endif

or:

#pragma once

if your portability requirements allow it.
Namespace pollution

Absolutely no:

typedef struct Matrix Matrix;

in global namespace.

Need:

typedef struct ml_matrix_t ml_matrix_t;

Everything starts with:

ml_

No exceptions.
3. The type system

This is where I would be extremely annoying.

A numerical library cannot casually use:

float
double
int
long

everywhere.

I want:

typedef float  ml_f32;
typedef double ml_f64;

typedef int32_t  ml_i32;
typedef uint32_t ml_u32;

from:

<stdint.h>

Why?

Because:

long

is not the same size everywhere.

Windows:

long = 32 bit

Linux x86_64:

long = 64 bit

This will destroy portability.
4. Error handling

This is a huge one.

Numerical libraries fail silently.

Example:

double x = sqrt(-1);

What happens?

NaN?

Error?

Exception?

Return code?

Document it.

I want:

typedef enum
{
    ML_SUCCESS = 0,

    ML_ERR_INVALID_ARGUMENT,
    ML_ERR_OVERFLOW,
    ML_ERR_UNDERFLOW,
    ML_ERR_DOMAIN,
    ML_ERR_MEMORY

} ml_status_t;

Functions:

Before:

double ml_sqrt(double x);

After:

ml_status_t ml_sqrt(
    double x,
    double *result
);

Why?

Because scientific code often cannot afford hidden failure.
